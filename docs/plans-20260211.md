# LitScribe 实施计划

## 项目概述

LitScribe 是一个基于 MCP 和多智能体架构的自动文献综述引擎，实现从文献检索、批判性阅读到智能综述生成的全流程自动化。

**核心创新点**：
1. 混合 AI 策略：本地模型（Qwen 3）处理高频任务，Claude API 处理复杂综合
2. 多智能体协作：Discovery → Critical Reading → Synthesis → (Peer Review)
3. 可追溯引用系统：每个声明追溯到原文段落、PDF 页码
4. 本地优先缓存：SQLite 持久化，支持断点续跑和增量更新

## 当前任务：Phase 11 — Per-Task Local LLM Model Routing

### Context

LitScribe 通过 `litellm` 调用 LLM（`agents/tools.py` 的 `call_llm()`），当前所有任务使用同一个模型（`LITELLM_MODEL` 环境变量，默认 `deepseek/deepseek-chat`）。唯一的路由是 `REASONING_TASK_TYPES = {"refinement"}` 会使用 `LITELLM_REASONING_MODEL`。

**问题**: 查询扩展、论文分析、综述生成等任务的复杂度差异大，但全部使用同一云端模型——成本高、隐私差。Ollama 原生支持 Metal/MPS 加速（Apple Silicon），litellm 已支持 `ollama/model` 格式，无需额外依赖。MLX 比 Ollama 快 30-50%（~230 vs ~150 tok/s），但需要 mlx-lm-server（OpenAI 兼容 API）。

**目标**: 支持 per-task 模型路由（通过 `~/.litscribe/config.yaml`），让用户为不同任务指定不同模型（本地或云端），本地失败时自动回退云端。

### 当前 LLM 调用链路

```
Agent 读取 model = state["llm_config"].get("model")
  → call_llm(prompt, model=model, task_type="xxx", agent_name="yyy")
    → _resolve_model(model, task_type)  # 决定最终模型
      → litellm.acompletion(model=resolved_model, ...)
```

`_resolve_model()` 当前优先级: explicit model > reasoning model (refinement) > Config.LITELLM_MODEL

#### Step 2: 修改 `_resolve_model()` (`src/agents/tools.py`)

新优先级链:

```
CLI --model (explicit)
  > user_config task_models[task_type]
    > user_config reasoning_model / env LITELLM_REASONING_MODEL (for REASONING_TASK_TYPES)
      > user_config default_model
        > env LITELLM_MODEL
```

```python
def _resolve_model(model, task_type=None):
    from utils.llm_routing import resolve_task_model, get_default_model, get_reasoning_model
    if model is not None:
        return model
    task_model = resolve_task_model(task_type)
    if task_model:
        return task_model
    reasoning = get_reasoning_model() or Config.LITELLM_REASONING_MODEL
    if task_type in REASONING_TASK_TYPES and reasoning:
        return reasoning
    return get_default_model() or Config.LITELLM_MODEL
```

### 12.1 订阅管理

**数据库扩展** (`src/cache/database.py`, Schema v5):
```sql
CREATE TABLE subscriptions (
    id INTEGER PRIMARY KEY,
    keywords TEXT,           -- JSON array of keywords
    sources TEXT,            -- JSON array: ["arxiv", "pubmed", "semantic_scholar"]
    frequency TEXT,          -- "daily" | "weekly"
    notify_methods TEXT,     -- JSON array: ["email", "webhook", "web"]

    last_run_at TIMESTAMP,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP

CREATE TABLE subscription_papers (
    id INTEGER PRIMARY KEY,
    subscription_id INTEGER REFERENCES subscriptions(id),
    paper_id TEXT,
    discovered_at TIMESTAMP,
    is_new BOOLEAN DEFAULT TRUE
);

CREATE TABLE notifications (
    id INTEGER PRIMARY KEY,
    subscription_id INTEGER,
    type TEXT,               -- "digest" | "review_suggestion"
    content TEXT,            -- JSON: summary, paper_count, etc.
    delivered_methods TEXT,  -- JSON: which methods succeeded
    created_at TIMESTAMP
);
```

### 存储方案

| DuckDB | 分析查询快 | 额外依赖 | ❌ |

### PDF 解析后端
| 后端 | 速度 | OCR | 稳定性 | 决策 |
|------|------|-----|--------|------|
| pymupdf4llm | 快 | 否 | 稳定 | ✅ 默认 |
| marker-pdf | 慢 | 是 | macOS MPS 不稳定 | 备选 |

### LLM 调用策略
| 任务 | 模型 | 理由 |
|------|------|------|
| 查询扩展 | Claude Haiku / 本地 | 简单任务，低成本 |
| 论文分析 | Claude Sonnet 4.5 | 平衡质量与成本 |
| 综述生成 | Claude Opus 4.5 | 复杂综合任务 |
| 大规模分析 | DeepSeek-R1 | 成本低，适合批量 |

### 速率限制
| API | 限制 | 缓解措施 |
|-----|------|---------|
| Semantic Scholar | 1 req/s | AsyncRateLimiter |
| PubMed | 3 req/s (无 key) | 缓存 + 批量请求 |
| arXiv | 无明确限制 | 礼貌性延迟 |
